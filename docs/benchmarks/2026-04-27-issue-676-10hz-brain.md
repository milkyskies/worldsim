# Tracy headless A/B — issue #676 10 Hz brain ticking

100-human stress headless run, seed 42, 2000 ticks, captured before and after
the brain-rate decoupling on the same machine. Methodology follows
`scripts/perf-snapshot.sh` (Tracy `--features profile-tracy`,
`tracy-csvexport`) but with `--humans 100` to surface brain-dominated cost.

## Run config

| field | value |
|---|---|
| date | 2026-04-27 |
| host | macOS arm64 (Apple silicon, M-series) |
| build | `cargo build --release --no-default-features --features profile-tracy` |
| invocation | `./target/release/worldsim --headless --game-defaults --humans 100 --seed 42 --ticks 2000` |
| pre commit  | `1b81f7b` (main, post-#368/#370/#587/#373/#367) |
| post commit | `574a65b` (`feature/#676.10hz-brain-ticking`) |

## Headline numbers

| metric | pre-#676 | post-#676 | Δ |
|---|---:|---:|---:|
| wall-clock (2000 ticks) | 46.9s | 31.0s | **-34%** |
| effective TPS | 43 | **64** | **+49% (crosses real-time)** |
| `FixedMain` per-tick mean | 22.99 ms | 15.06 ms | **-7.93 ms (-34%)** |
| `arbitrate_every_tick` per-tick mean | 9.88 ms | 1.79 ms | **-82%** |

`arbitrate_every_tick` now fires every 6th tick instead of every tick (333
calls vs 2000 over the run), so per-call cost rises from 158 μs to 419 μs
because each call drains six ticks of accumulated wakeups — but per-tick
cost (the real metric) drops 5.5×.

The 60 TPS real-time target was the issue's primary acceptance criterion;
baseline missed it (43 TPS) and the branch crosses it comfortably (64 TPS).

## Notes

- The issue projected ~7.07 ms/tick saving from gating both
  `arbitrate_every_tick` and `update_rational_planning` at 10 Hz. Measured:
  ~7.93 ms/tick saving from gating only arbitration plus the heavy GOAP
  search step internally — `update_rational_planning`'s plan-verification
  loop stays at 60 Hz to consume single-pass `SimEvent`s before Bevy's
  message buffer clears them.
- The planner cost the issue projected (~1.5 ms/tick at 100 humans) was
  much smaller in practice (~0.29 ms/tick), so leaving it at 60 Hz costs
  almost nothing — the SimEvent-correctness win is essentially free.
- `react_to_danger` is now the new dominant cost on this config (~5.6
  ms/tick, ~37% of `FixedMain`). Next 100-human perf work should target it.
